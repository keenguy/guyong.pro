Git index notes

git的核心在于commits, 大多数命令都是在操作 Commits. 每个 commit 相当于整个项目的一个快照（snapshot）。
Use gitk to understand git这篇文章详细的讲解了commit tree在一些常用操作下的变化。

基本概念及操作

  1. working directory
    git所维护的本地工作目录， 所有的本地操作都是在操作该目录。里面包括.git/ 以及用户文件，可能包括.gitignore文件。
  • git init---- setup git
  1. remote repository
    远程仓库，根据传递方向分为pull和push两种。pull是用来将远程仓库下载到本地工作目录，而push则相反。传递方式一般有https和ssh两种，前者用密码来鉴定权限，后者用密钥。
  • git remote add git remote rm ---- add/remove remote repository
  • git clone ---- download remote repository and assosiciate it (with default name origin).
  • git fetch git push git pull

  1. track,stage,commit
    Working directory下的文件有四种类型，untracked, unmodified, modified, stated. git的基本工作原理是对项目文件‘拍照’（snapshot)。 命令git commit会将staged files‘拍’成一个snapshot. Untracked files是指那些未被git管理的文件，unmodified files是指存在上一个snapshot中的文件，modified是指上一个快照后有改动的文件，staged files是指“提交”到staging area中的文件。git add可以将untracked或modified files “提交” 到staging area 中，以备拍照使用。由于每一个commit对应一个snapshot, 为了方便，下面有时会用commit来表示snapshot.

    Figure 1. The lifecycle of the status of your files.

  2. commit tree, branch, HEAD
    git进行版本控制的核心在于控制commits, 为了性能考虑各个commit不是独立的（大概是一个增量存储？）。因此各个commit会以树形（或图形）结构存储。

    初始时只有一个commit节点（Initial commit), 如果只是一步步commit的话，该commit tree会是一条直线，不会有分叉。根据需要，git中的commit tree可能会出现多个分支，git 用branch 来标记这些分叉（始终指向该分叉上最新的commit节点）。 HEAD是一个特殊的标签（ref)，它始终指向当前的分支（current branch)，可以看作该branch的一个别名 。该branch会随着git commit命令往前移动，也就是说它会始终指向当前分支的最新的commit.
    git branch branch1---- 从HEAD指向的commit上新建一个分支，本质是给该commit加一个新的标签（索引）。可以看作树上在commit节点上长出了一个(空的)新分支.
    git checkout branch1 ---- HEAD跳向 branch1.

  3. merge和rebase
    merge是指将两个commit节点merge成一个新的节点。假设HEAD指向master, git merge branch1 会将master与branch1 merge成一个新的节点，并且master会指向这个新的节点。（可以理解为用branch1指向的commit 来更新master指向的commit）.
    Fast forward: 若存在master到branch1的直线（即两个节点merge成的节点内容与branch1节点的内容相同），则该操作仅仅是把master的指向改变，从指向有的是commit改成指向branch1指向的commit.
    Real merge: 若不存在master到branch1的直线，则该操作会生成一个新的基于master和branch1的commit. git会记录两个分支历史。
    若不想要git记录两个分支，可以用rebase来实现：git checkout branch1 git rebase master

  4. push, fetch, pull, upstream, downstream
    git push origin master 把本地的master分支merge到origin上去。
    git fetch origin 把origin上的新的branch加到当前的commit tree中，并不会进行merge.
    git pull 相当于先fetch后merge

命令详解

  1. git status, git status -s
    outpus git comprehensive/short stuatus info.
  2. git diff
    This command compares what is in your working directory with what is in your staging area. It’s important to note that git diff by itself doesn’t show all changes made since your last commit – only changes that are still unstaged.
    git diff --staged
    This command compares your staged changes to your last commit.
    git diff --cached
    Used to see what you’ve staged so far (–staged and --cached are synonyms).
  3. git commit
    git commit -m "your commit message"
    git commit -a skip the staging process, commit all tracked files.
  4. git remove
    remove local files and stage the removal.
    git rm --cached unstage files.
  5. git mv file_from file_to
    It’s equivalent to: 
    $ mv README.md README
    $ git rm README.md
    $ git add README

Simply rename a file wouldn’t be detected by git.

  1. git log
  2. undoing things
    git commit --amend ---- amend last commit.
    git reset HEAD <file> ---- Unstaging a Staged File
    git checkout -- <file> ---- Unmodifying a Modified File
